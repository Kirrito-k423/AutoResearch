# Phase 10 Plan 01: 8 个 skill workflow + Archon runtime adapter

Plan 01 先把“每个 skill 都能被 Archon 单独触发”这件事做实：补一个 repo-local Archon runtime adapter，把现有 Python 入口包成 `.archon/scripts/` 可调用的 deterministic script，再给 8 个 skill 各写一份 workflow YAML。

## Tasks

1. **`autoresearch/archon/runtime.py` + `__init__.py`**
   - 新建 Phase 10 适配层，职责只做 Archon runtime -> 现有 `run_*` 入口映射
   - 统一解析环境变量：
     - `AR_CONFIG_PATH`
     - `AR_SERVER`
     - `AR_LIB`
     - `AR_TIMEOUT`
     - `AR_PUSHGATEWAY_URL`
     - `AR_RUN_ID`
   - 默认值与 D-58 对齐：
     - `config/config.yaml`
     - config 第一台 server
     - `verl`
     - `60`
     - `http://127.0.0.1:17891`
   - 提供 8 个薄封装函数，例如：
     - `run_skill_01()` -> `run_validate(..., as_json=True)`
     - `run_skill_02()` -> `run_start()` 后接 `run_status(as_json=True)`
     - `run_skill_03()` -> `hw.run_probe(...)`
     - `run_skill_05()` -> `run_reach_test(...)`
     - `run_skill_07()` -> `run_collect(...)`
     - `run_skill_08()` -> 从 `AR_RUN_ID` 或 `$ARTIFACTS_DIR/collect-result.json` 取 run_id，再 `run_render(...)`
   - 对需要 handoff 的步骤把结构化 payload 写入 `$ARTIFACTS_DIR`，至少包含：
     - `stack-state.json`
     - `collect-result.json`
     - `report-result.json`

2. **`.archon/scripts/ar-skill-01.py` .. `ar-skill-08.py`**
   - 每个脚本只做一件事：调 `autoresearch.archon.runtime` 中对应函数，并把最终 JSON 打到 stdout
   - 这些脚本要能被 Archon `script:` + `runtime: uv` 直接运行
   - 不在脚本里复制业务逻辑；脚本只是 Archon named-script adapter

3. **`.archon/workflows/ar-skill-01.yaml` .. `ar-skill-08.yaml`**
   - Skill 01..05 / 08 用 deterministic script node 即可
   - Skill 06 / 07 用“setup script -> `loop:` -> summary script”模式：
     - 06: 逐个处理 `verl` / `veomni`
     - 07: 有上限的自动重试 collect
   - workflow 名称固定为：
     - `ar-skill-01`
     - `ar-skill-02`
     - ...
     - `ar-skill-08`
   - YAML 里不硬编码 provider/model，保持对用户本机 Archon 配置透明

4. **STACK loop 细化**
   - `ar-skill-06.yaml` 的 setup 节点负责初始化 `stack-state.json`
   - loop 节点每次只处理一个 lib，并把结果累积到 state 文件
   - summary 节点汇总两次检查结果，按整体 pass/fail 退出
   - `until_bash` 读取 state 文件判定是否“所有 libs 都处理完”

5. **COLL loop 细化**
   - `ar-skill-07.yaml` 的 setup 节点初始化 attempt 计数与 state 文件
   - loop 节点每轮跑一次 `run_collect()`；失败时记录 payload 和错误
   - 成功后把 `run_id` / manifest / artifact 路径写到 `collect-result.json`
   - `until_bash` 在“成功”或“达到最大 attempts”时退出

6. **测试**
   - `tests/test_archon_runtime.py`
     - 环境变量默认值解析
     - config 第一台 server 解析
     - collect -> report 的 handoff 文件读取
     - services happy-path 聚合输出
   - 不要求在单测里真正启动 Archon；重点验证 Python 适配层 contract

## 风险

- **R1 script node import 路径问题**：`.archon/scripts/*.py` 运行时必须能 import repo 内包；若 `uv run <path>` 的工作目录与预期不同，需要在 runtime adapter 内显式处理 repo root。
- **R2 现有 `run_*` 返回风格不完全一致**：有的返回 exit code，有的返回 `(exit_code, payload)`；runtime adapter 需要统一 JSON contract。
- **R3 collect -> report 交接脆弱**：必须优先用 `$ARTIFACTS_DIR/collect-result.json` 做显式 handoff，不能指望 prompt 文本或手工复制 run_id。
- **R4 loop 节点不支持 script/batch 混用**：06/07 的迭代逻辑要通过 state 文件 + `until_bash` 做 deterministic stop，避免把退出条件写死在自由文本里。
