# Phase 10 Plan 02: 主 workflow `ar-min-loop.yaml`

Plan 02 把 8 个 skill 拼成一次最小闭环，让 Archon 不只是“能单独点 skill”，而是能真正从 config 一路走到 report。

## Tasks

1. **`.archon/workflows/ar-min-loop.yaml`**
   - 定义主 workflow `ar-min-loop`
   - 节点顺序固定：
     1. config validate
     2. services ensure + status
     3. hw probe
     4. net probe
     5. reach test
     6. stack loop
     7. collect loop
     8. report render
   - 依赖关系明确使用 `depends_on`
   - 默认在 live checkout 运行（验证阶段用 `--no-worktree`），不在 YAML 里自己管理 worktree

2. **重用 10-01 的 skill adapter，而不是递归调用 `archon workflow run`**
   - 主 workflow 节点复用同一套 `.archon/scripts/*.py` / `autoresearch.archon.runtime`
   - 不在 workflow 内部再 shell 出第二层 `archon workflow run ar-skill-XX`；避免递归运行、路径混乱和状态分叉
   - 对 06 / 07，直接在主 workflow 里复用同样的 setup/loop/summary 结构

3. **collect -> report 交接**
   - collect summary 节点把 `collect-result.json` 写到同一次 Archon run 的 `$ARTIFACTS_DIR`
   - report 节点从该文件解析 `run_id`
   - 如果 `collect-result.json` 缺失或 `ok=false`，report 节点应明确失败，而不是生成空报告

4. **workflow-level 可观察性**
   - 主 workflow 的 description 要直接表明“8-skill 最小循环”
   - 节点命名要可读，便于在 Archon Web UI 中扫到当前卡在哪一步
   - 对关键失败（stack/collect/report）保持错误消息直白，避免 UI 上只剩 exit code

5. **测试**
   - `tests/test_archon_workflows.py`
     - 解析 9 份 YAML，检查：
       - workflow 名称唯一
       - `ar-min-loop` 存在并含完整 skill 链
       - `ar-skill-06.yaml` / `ar-skill-07.yaml` 包含 `loop:`
       - 主 workflow 的 `report` 节点在 collect 之后
     - 这是 repo 内的静态 contract test，不替代 `archon validate workflows`

## 风险

- **R1 主 workflow 与 skill workflow 漂移**：如果主 workflow 单独抄一份逻辑，很快会与 skill workflow 分叉；所以必须共用 runtime adapter 和命名规则。
- **R2 report 节点过早执行**：collect 结果必须通过 artifact file 明确交接，避免因为节点顺序或隐式状态导致 report 读不到 run_id。
- **R3 YAML 只“看起来对”**：静态测试只能保 schema / depends_on / loop 形状，真实可运行性仍要在 Plan 03 里靠 `archon validate` + `workflow run` 验证。
