# Phase 06 Plan 03: reach test --all 并发 + 失败诊断

`autoresearch reach test --all` 走并发, 4 台真机 UAT 一次性出报告. 复用 Phase 4 hw probe 的并发模式.

## Tasks

1. **`test_all_servers` 实现** (`autoresearch/reach/tester.py`)
   - 同 04-03 模式: `ThreadPoolExecutor(max_workers=3)`
   - 每 worker 调 `test_server_reach`
   - 失败隔离: 单 worker raise/return fail 不影响其他
   - 结果按 config.servers 顺序排
2. **CLI `--all` 接入** (`autoresearch/cli.py`)
   - `--all` 与 `--server` 互斥, 给 click 错误
   - 输出 JSON: `{"ok": bool, "severity": str, "data": {"results": {...}, "total": N, "passed": N, "failed": N, "passed_servers": [...], "failed_servers": [...]}}`
3. **失败诊断增强**
   - 探测失败自动调 `run_tunnel_ensure` 再试一次 (bound 8s); 仍失败返回 error 含 D-37.C2 文案
4. **测试** (`tests/test_reach_tester.py`)
   - mock 3 worker, fixture:
     - 4 机器全 PASS → overall ok=True
     - 1 机器 fail → 其他 3 PASS, overall fail, failed_servers 含 1 项
     - 1 机器 fail → 自动重试 1 次仍 fail (mock 始终返 fail) → error 含 D-37.C2 文案
     - 顺序: config 顺序与 result 顺序一致
   - click CliRunner 测 `--all` happy + `--all --server` 互斥
5. **UAT** (`06-03-SUMMARY.md`):
   - 4 台真机 `autoresearch reach test --all`, 期望 passed=4, failed=0 (基于 06-02 已 OK)
   - 记录每台: tunnel 建立耗时, wandb /healthz 200 latency, pushgateway push latency
   - 至少 1 台确认 Prometheus 端能 query 到 autoresearch_reach_test metric

## 风险

- **D-37 重试 + 并发交互**: 同一 server 多次 run_tunnel_ensure 可能在并发下撞 (singleton tunnel state); 加锁或顺序化重试
