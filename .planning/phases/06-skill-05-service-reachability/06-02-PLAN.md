# Phase 06 Plan 02: reach test 单机实现 (wandb + pushgateway)

落地 `autoresearch reach test --server X`, 验证远程服务器通过隧道访问本机 wandb (8080) 和 pushgateway (9091).
本 plan 不做并发 (`--all` 是 06-03).

## Tasks

1. **`autoresearch/reach/` 模块骨架**
   - `__init__.py` (空) + `models.py` (ReachResult TypedDict, ReachCheck TypedDict) + `tester.py` (核心)
2. **`autoresearch/reach/tester.py` 核心**
   - `test_server_reach(server_name, config_path, local_proxy_url, lang) -> ReachResult`
   - 步骤:
     a) `_ensure_tunnel_for_reach()` 调 `run_tunnel_ensure()` 两次 (端口 17890 + 17891)
     b) 远程走 `curl -fsS -m 8 http://127.0.0.1:17890/healthz` → parse body 验 `state == available`
     c) 远程走 `curl -fsS -X PUT -m 8 --data-binary '...' http://127.0.0.1:17891/metrics/job/...`
     d) 汇总 wandb ok / pushgateway ok → 整体 severity
   - 失败诊断: 探测失败 → 返回 `error` 含 D-37.C2 文案
3. **CLI 接入** (`autoresearch/cli.py`)
   - `@reach.group(name="reach")` 新 CLI group
   - `@reach.command("test")` 接 `--server`, `--config`, `--lang`
   - 输出 JSON + 走 `__AR_PROGRESS__` (用 `workspace_core.progress.emit_progress`)
4. **测试** (`tests/test_reach_tester.py` + `tests/test_reach_cli.py`)
   - mock SSHClient, fixture 覆盖:
     - happy: wandb 200 + pushgateway 200 → ok=True
     - wandb fail → 走 _ensure_tunnel_for_reach 重试 → 仍 fail → error 含 D-37.C2 文案
     - pushgateway push fail → warn 不 fail (D-38.D4)
   - click CliRunner 测 CLI happy path
5. **UAT** (`06-02-SUMMARY.md`):
   - 4 台真机: 跑 `autoresearch reach test --server A2-AK-225` 等, 记录每台结果
   - 至少 1 台验证 pushgateway metric 真的落到 Prometheus (`curl -G http://127.0.0.1:9090/api/v1/query --data-urlencode 'query=autoresearch_reach_test'`)

## 风险

- **D-37.C1 双 tunnel**: 17890 + 17891 同步起, run_tunnel_ensure 串行不会撞; 但要 bound 8s
- **pushgateway body 注入**: metric body 包含 server name, 必须 sanitized (避免在 stderr log 泄露 host)
- **remote_proxy_port 参数化**: 默认 17890/17891, 但 CLI 允许 override 用于本地多 tunnel 调试
